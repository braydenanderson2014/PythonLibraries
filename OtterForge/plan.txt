Name : OtterForge

OtterForge Build Plan

Objective
Build a Python application named OtterForge that helps users scan projects, understand project structure, build Python executables and packages, and create installers on Windows, Linux, and macOS. The product must be CLI-first, must also provide a PyQt desktop UI, and must preserve persistent memory so users can reuse previous project settings, tool selections, arguments, project descriptions, and build history.

This document is written for AI implementation agents. Treat it as a build specification, not as marketing copy.

Core Product Definition
OtterForge is a build orchestration tool for Python applications and Python-based software delivery workflows. It also supports building executables and packages from C-style compiled languages such as C, C++, Rust, and Go through the same adapter and manifest-controlled module system.

OtterForge is not only a wrapper around PyInstaller. It must support multiple executable builders, packagers, and installer generators through a plugin or adapter system. PyInstaller is the default and must be the first fully implemented adapter. All other builders must be supported through exactly the same adapter architecture so adding any new tool never requires changes to the core system.

OtterForge must also support package-oriented workflows where the desired output is not only an executable. It should be able to orchestrate common packaging tools such as build, setuptools, hatchling, poetry, briefcase, or other relevant tools when those tools are present and appropriate for the project.

OtterForge must also support installer creation workflows. Installer generation should be handled through dedicated adapters for tools appropriate to the target platform, such as Inno Setup, NSIS, WiX, pkgbuild or productbuild, DMG tooling, AppImage tooling, or similar common installer systems.

OtterForge must be able to work with both automatically discovered project structure and a user-supplied JSON description of project layout. The JSON description must allow OtterForge to understand folders, entry points, assets, dependencies, packaging intent, installer intent, and platform-specific rules for projects that do not follow common conventions.

OtterForge must have three interfaces:
1. A CLI for scripting, automation, CI, and power users.
2. A PyQt GUI that uses the same backend services as the CLI.
3. A Python API layer so future automation or integrations can call the engine directly.

Module Manifest System
OtterForge uses a module manifest to track the availability of every optional capability at runtime. The manifest is the authoritative record of what is present, what is installed, and what is enabled. The core application must never crash, produce an error dialog, or raise an unhandled exception because an optional module, adapter, or tool is absent.

Manifest structure:
The manifest is a JSON file named otterforge.manifest.json stored in OtterForge's data directory. It is generated or refreshed at startup and on demand via otterforge manifest refresh.

Each entry in the manifest covers one capability. Entry fields:
1. id: Unique string identifier for the capability, e.g. builder.pyinstaller or lang.rust.
2. category: One of builder, packager, installer, language, service, plugin.
3. display_name: Human-readable name.
4. description: One-line description of what the capability provides.
5. available: Boolean. True if the required tool or module is found on the current machine.
6. enabled: Boolean. Allows the user to disable a capability even if it is available.
7. requires: List of external tool names or Python packages required for this capability.
8. missing: List of requirements that were not found during the last refresh.
9. platform_support: List of platforms this capability works on, e.g. windows, linux, macos.
10. adapter_module: Dotted Python module path to the adapter class.
11. version_detected: The version string of the detected tool, or null if not found.
12. last_checked: ISO 8601 timestamp of the last availability check.

Manifest behavior rules:
1. At startup OtterForge calls manifest.refresh() which probes each registered capability for availability without importing optional dependencies eagerly.
2. Any registry, the CLI listings, and the GUI builder selector all read from the manifest rather than probing tools directly.
3. A capability that is unavailable is shown in the GUI as grayed out with a tooltip listing the missing requirements.
4. A capability that fails to load at runtime logs the failure, marks the manifest entry as available=false, and continues. The rest of the application is unaffected.
5. The user can view the full manifest at any time with otterforge manifest show.
6. otterforge manifest refresh re-probes all capabilities and updates the manifest.
7. Individual entries can be disabled with otterforge manifest disable <id> and re-enabled with otterforge manifest enable <id>.
8. The manifest is also the hook point for plugins: when a plugin is installed, it writes its entries into the manifest and they are probed on the next refresh.
9. The manifest must be human-readable and hand-editable in emergencies. Validate it on load and surface clear parse errors.

Relationship to the plugin system:
The plugin loader and the manifest system are complementary. The plugin loader discovers adapter classes. The manifest system tracks their real-world availability. Every adapter discovered by any loader, built-in or plugin, must have a manifest entry.

C-Style and Compiled Language Support
OtterForge supports building executables from compiled languages through dedicated language adapters. These adapters are first-class modular addons governed by the manifest system. They are completely optional and the core Python build workflows must work without them.

Supported compiled languages:

C and C++
Adapter builds using the system compiler toolchain via a normalized invocation. On Windows the adapter looks for MSVC (cl.exe) or MinGW (gcc, g++). On Linux and macOS it uses gcc or clang. The adapter normalizes common options such as source files, output name, include paths, library paths, linked libraries, optimization level, debug symbols, and standard version such as C17 or C++20. CMake is treated as an optional higher-level front-end: if a CMakeLists.txt is present and cmake is available, the adapter generates and runs the appropriate cmake and cmake --build commands instead of invoking the compiler directly. A Makefile front-end is also provided for projects using make.

Rust
Adapter builds using cargo. The normalized options are the package or workspace path, the binary target name, the profile (debug or release), target triple for cross-compilation, feature flags, and extra cargo build arguments. The adapter checks for cargo in PATH. If cargo is absent the manifest marks the Rust adapter as unavailable.

Go
Adapter builds using the go toolchain. Normalized options are the main package path, the output binary name, GOOS and GOARCH for cross-compilation, build tags, CGO enabled or disabled, and extra go build arguments. The adapter checks for go in PATH. If go is absent the manifest marks the Go adapter as unavailable.

Language adapter design rules:
1. Every language adapter implements the same Tool Adapter Contract as Python executable builders.
2. Each language adapter has its own manifest entry with category language.
3. The CommandTranslator is not involved; language adapters produce their own command directly because there is no Python-specific memory config to translate.
4. Language adapters expose the same dry-run mode, raw pass-through arguments, and raw argument delimiter that Python builders use.
5. A build request for a compiled language must carry language-specific fields in an extension dict rather than sharing the Python-centric BuildRequest fields. A base LanguageBuildRequest model is defined separately.
6. Language adapters integrate with all shared services: pre and post hooks, code signing, artifact checksums, size analysis, git integration, and build notifications.
7. The GUI shows language adapters in the builder selector grouped separately from Python builders, labeled as Native Language Adapters.
8. The CLI accepts otterforge build --lang c, --lang cpp, --lang rust, or --lang go to invoke a language adapter.
9. Language adapters are disabled by default and must be explicitly enabled by the user or enabled automatically when the required toolchain is detected and the user has run otterforge manifest refresh.
10. Each language adapter is implemented in otterforge/langs/ as a standalone module: c_cpp.py, rust.py, go.py. Each must implement the standard Tool Adapter Contract. Each is optional at runtime and must degrade gracefully when not installed. The module manifest system (described in a later section) is the single source of truth for which adapters are present, available, and enabled on the current machine.

PyInstaller
PyInstaller is the default builder and the first one implemented. It works on Windows, Linux, and macOS. It produces standalone executables from Python scripts. Its key options include onefile or onedir output mode, noconsole or console, hidden imports, additional data files, UPX compression, splash screen support, icon embedding, spec file support, and a clean flag. OtterForge should normalize all these as first-class fields. Raw pass-through must also be supported for any PyInstaller flag not normalized.

Nuitka
Nuitka is a Python-to-C compiler. It works on Windows, Linux, and macOS. It produces fast native executables by compiling Python to C and then to machine code. It requires a C compiler such as MSVC on Windows, GCC, or Clang on Linux and macOS. Key options include standalone mode, onefile mode, module mode, plugin system, follow imports, show progress, windows icon, windows company name, and LTO. These should be normalized where equivalent to PyInstaller options. Nuitka has a rich plugin system; raw pass-through is important here.

cx_Freeze
cx_Freeze works on Windows, Linux, and macOS. It freezes Python scripts into executables by bundling the interpreter. Its configuration can be expressed through setup scripts or command-line options. Key options include target name, base class for windowed apps, include files, excludes, includes, zip include packages, and build directory. cx_Freeze supports pyproject.toml configuration. OtterForge should support both command-line and a generated setup hook approach.

py2app
py2app is macOS-only. It creates macOS application bundles in .app format. It can also produce .dmg files. Key options include the main script, name, icon, plist customization, packages to include, frameworks, dyld includes, and recipe system. OtterForge must surface py2app only when the host is macOS and only invoke it on macOS CI runners.

py2exe
py2exe is Windows-only. It bundles Python programs into Windows executables. Key options include console and windows targets, optimization level, bundle level, and dist directory. OtterForge must surface py2exe only when the host is Windows and only invoke it on Windows CI runners.

briefcase
briefcase is cross-platform but produces platform-native packages rather than only executables. On Windows it produces MSI installers. On macOS it produces .app and .dmg. On Linux it produces AppImage, Flatpak, or Snap. It uses a pyproject.toml configuration section. Key options include app identifier, display name, version, author, icon, and platform-specific extras. Briefcase bridges the builder and installer categories and should be registered in both the builders and the installer registries.

shiv
shiv creates self-contained Python zipapps. It targets platforms where Python is already installed. It is not a standalone executable builder but is useful for CLI tool distribution. Key options include the entry point, output file, and pip constraints. OtterForge should register shiv as an alternative builder with a clear note that it requires Python on the target machine.

zipapp
zipapp is part of the Python standard library. It creates a .pyz archive that can be run with any compatible Python interpreter. It is the simplest distribution format and requires no extra dependencies. OtterForge should register zipapp as a built-in builder requiring zero tool installation.

Platform Availability Matrix
The platform availability matrix describes which builder runs on which host platform.

1. PyInstaller: Windows, Linux, macOS.
2. Nuitka: Windows, Linux, macOS.
3. cx_Freeze: Windows, Linux, macOS.
4. py2app: macOS only.
5. py2exe: Windows only.
6. briefcase: Windows, Linux, macOS as host, but each host produces only its own platform target.
7. shiv: Windows, Linux, macOS where Python is on the target.
8. zipapp: Windows, Linux, macOS where Python is on the target.

The system must read the platform availability matrix at build time and suppress or warn for builders that cannot run on the current host.

Hard Requirements
1. The program must run on Windows, Linux, and macOS.
2. The program must be able to invoke common Python executable builders, packagers, and installer generators if they are installed in the active environment.
3. The program must support arbitrary tool-specific flags and arguments.
4. The program must remember prior builds, packaging settings, installer settings, discovered entry points, output locations, project descriptions, and reusable profiles.
5. The CLI and GUI must share one backend so behavior stays consistent.
6. The system must be designed for local use and CI use, including GitHub Actions or similar cloud runners.
7. The plan must assume incremental delivery. Build the architecture first, then add builders.
8. The system must be able to understand nonstandard project layouts from a JSON structure supplied by the user.
9. The system must support project scanning for projects that are script-based, package-based, GUI-based, service-based, or multi-entrypoint.
10. The scanner must support scope-based discovery for files, folders, projects, and user-defined groups, and must classify assets and documentation by file type.

Important Technical Constraint
Do not design this as if one machine can always produce native executables for every other operating system.

Accurate rule:
1. OtterForge must run on any major platform.
2. OtterForge should build native executables on the current host platform.
3. OtterForge should be able to orchestrate multi-platform builds through CI or remote runners.
4. OtterForge should not promise universal cross-compilation when the underlying builder does not support it.

Example:
1. A Linux host should be able to build Linux targets locally.
2. A Windows host should be able to build Windows targets locally.
3. Multi-platform output should be handled through a matrix build workflow when required.

Design Principles
1. CLI-first architecture.
2. Backend services are framework-agnostic.
3. Tool support is adapter-based, not hard-coded into one command.
4. Common options should be normalized where useful, but raw pass-through must always be available.
5. Persistent memory should be explicit, inspectable, and editable.
6. Every operation should be traceable through logs and stored build records.
7. The GUI should be a thin client over the same command or service layer.
8. Auto-discovery and user-provided project schemas must both be first-class inputs.

Tool Strategy
The architecture must separate normalized arguments from raw pass-through arguments.

Required behavior:
1. Common options such as entry script, app name, onefile or onedir, console or noconsole, icon, splash, clean build, output directory, hidden imports, data files, extra paths, package metadata, and installer metadata should be represented in a normalized internal model.
2. Every tool adapter must also accept raw arguments exactly as the user entered them.
3. The CLI must support a safe delimiter so everything after that delimiter is passed directly to the selected tool.
4. The GUI must expose both structured fields and an advanced raw arguments field.
5. The orchestration layer must allow multi-step workflows such as scan -> build executable -> package artifacts -> generate installer.

Recommended CLI pattern:
1. Use OtterForge flags before a delimiter.
2. Pass builder-native flags after the delimiter.

Example shape:
otterforge build --builder pyinstaller --entry main.py --profile desktop -- --collect-all PyQt6 --hidden-import pkg_resources

This design is required because no static hand-maintained option list will stay complete for every tool version.

MCP Support Requirements
OtterForge must support Model Context Protocol (MCP) so AI agents can use OtterForge through a structured tool interface instead of shell scraping.

MCP server scope:
1. Implement an MCP server mode that exposes selected OtterForge capabilities as MCP tools.
2. The MCP server is optional and must not be required for normal CLI or GUI usage.
3. The MCP server must use the same backend services as CLI and GUI so behavior is identical.
4. stdio transport is required first. Network transport is optional and can be added later.

MCP tool exposure rules:
1. Expose read tools by default, such as scan, list builders, inspect tools, list artifacts, and read build history.
2. Expose write or execution tools only when explicitly enabled by policy, such as build, package, installer create, and organize apply.
3. Every exposed MCP tool must map to an existing service method, not duplicate business logic.
4. MCP tool availability must honor the module manifest and backend capability checks.
5. If a capability is unavailable or disabled in manifest, the MCP tool must be hidden or return a structured unavailable response.

MCP security and safety rules:
1. Default MCP mode is read-only.
2. Destructive or mutation operations must require explicit enablement in config.
3. All MCP requests must be validated with the same request models used by CLI and GUI.
4. Secrets in responses must be redacted (tokens, certificate paths with passwords, signing secrets, webhook credentials).
5. MCP calls must be fully logged with session id, tool id, timestamp, and result status for auditability.

MCP configuration:
1. Add an MCP config section that controls enabled state, transport, exposed tools allowlist, and mutation policy.
2. The allowlist must support exposing only a subset of tools in sensitive environments.
3. The MCP server should publish metadata describing available tools and required arguments.

MCP non-goal for v1:
1. Do not build custom AI logic inside OtterForge itself. OtterForge provides tools; external AI clients provide reasoning.

Command Translation Layer
This is the central mechanism that makes OtterForge work. The user never constructs tool commands by hand. Instead, the user specifies settings in OtterForge terms and the translation layer converts those settings into the exact tool-specific command at build time.

How it works, step by step:
1. The user sets options through CLI flags, the GUI, or a project schema file.
2. OtterForge writes all settings into the active memory file immediately.
3. When the user calls any build, package, or installer command, OtterForge reads settings from memory.
4. The CommandTranslator service loads the correct adapter for the selected tool.
5. The adapter receives a BuildRequest, PackageRequest, or InstallerRequest populated from memory.
6. The adapter translates that request into the final tool-specific argument list.
7. OtterForge prints the translated command to the console before executing.
8. The executor runs the command with subprocess using that argument list.
9. OtterForge writes the result, the final command, and environment metadata back into memory.

The CommandTranslator must be implemented as an explicit, testable service class. It must not be embedded inside the CLI parser, the GUI, or any adapter. It sits between the stored settings and the adapter.

CommandTranslator responsibilities:
1. Read the active project's settings from memory.
2. Merge in any run-time overrides from the CLI or GUI.
3. Apply the override precedence order.
4. Construct the appropriate request model.
5. Resolve which adapter to use.
6. Call the adapter's build_command() method to produce the argument list.
7. Append raw pass-through args from the delimiter section.
8. Return the final command as a list of strings.

Override precedence applied by the translator:
Run-time CLI or GUI override > active profile settings > project settings > global user settings > adapter defaults

The translator must produce identical output whether called from the CLI, the GUI, or the Python API. This is the guarantee that all three interfaces behave the same way.

Dry-run requirement:
When dry_run is true, the translator must still produce and print the full translated command but must not invoke the executor. This allows users to preview the exact tool command before committing to a build.

Examples of what the translator produces for different builders given the same normalized input:

Input settings in memory:
  entry: main.py
  name: MyApp
  mode: onefile
  console: false
  icon: assets/icon.ico
  builder: pyinstaller

PyInstaller translation output:
  pyinstaller --onefile --noconsole --name MyApp --icon assets/icon.ico main.py

Nuitka translation output:
  python -m nuitka --standalone --onefile --disable-console --output-filename=MyApp --windows-icon-from-ico=assets/icon.ico main.py

cx_Freeze translation output:
  cxfreeze main.py --target-name=MyApp --base-name=Win32GUI --target-dir=dist

The actual flag names differ per tool. The translation layer absorbs that difference entirely. The user always works in OtterForge terms.

Memory Requirements
OtterForge memory is the single source of truth for all active settings. The CommandTranslator reads from memory at build time. Settings are never passed only as transient CLI arguments and then discarded; they are written to memory first and then read back by the translator.

Memory storage format: The user must be able to choose a memory backend: JSON files or SQL database.

Required backend behavior:
1. JSON backend must remain supported.
2. SQL backend must be supported as a first-class option using SQLite at minimum.
3. All backend access must go through an abstract MemoryStore interface so core services are backend-agnostic.
4. Backend choice can be set globally and overridden per project.
5. Switching backend must not lose data; migration tooling is required.

Memory must store:
1. Known projects and their paths.
2. Active project selection.
3. Active builder, packager, and installer selection per project.
4. All normalized build settings per project.
5. All raw pass-through arguments per project and per tool.
6. Saved named profiles.
7. Last successful build per project and per target platform.
8. Artifact paths per build.
9. The exact final translated command from every past run.
10. Environment snapshot from every past run.
11. Build logs or build log file references.
12. Tool usage history.
13. Imported and generated project schema file paths.
14. Installer generation history.
15. Global user preferences.
16. GUI state in a separate key so it never contaminates project settings.

Memory write rules:
1. Every time a user sets or changes any option, write it to memory immediately.
2. Never require the user to explicitly save before a build.
3. After every successful or failed build, append a record to build history.
4. Every build record must include the final command list, timestamps, builder name, platform, and exit code.

Memory clear commands:
The system must provide the following wipe operations, each scoped precisely:
1. otterforge memory clear --settings: Clears all active project settings and builder selection. Resets to defaults for the current project only.
2. otterforge memory clear --history: Clears all build history records. Does not affect settings.
3. otterforge memory clear --profiles: Removes all saved named profiles.
4. otterforge memory clear --all: Wipes all memory files completely and resets to a blank state.
5. otterforge memory clear --project <name>: Removes all memory associated with a specific project without affecting others.

Each clear operation must prompt the user for confirmation unless a --yes flag is provided.

The GUI must expose these same clear operations in a Settings or Memory management screen.

Suggested memory files:
1. user_settings.json
2. projects.json
3. build_history.json
4. profiles.json

Memory backend selection requirements:
1. Backend can be selected at init and changed later.
2. Default backend is JSON for zero-setup local usage.
3. SQL backend stores equivalent records in normalized tables.
4. A migration command must copy all memory data between backends safely.
5. If SQL backend is unavailable, OtterForge must fail with a clear error and recovery instructions.
6. GUI must expose backend selection, migration status, and a last-migration timestamp.

Project Discovery Requirements
The scan process should inspect a Python project and suggest buildable entry points, packaging strategies, and installer candidates.

Scan should detect:
1. Scanner scopes: files, folders, projects, and logical groups.
2. Candidate main scripts and executable entry points.
3. Existing spec files.
4. Existing pyproject.toml, setup.py, requirements files, and virtual environments.
5. GUI frameworks such as PyQt, Tkinter, Kivy, or others when obvious.
6. Asset folders and likely icon files.
7. Icon candidates by extension, including .ico, .icns, .png, .svg, and .bmp.
8. Splash screen candidates by extension, including .png, .jpg, .jpeg, .ico, .bmp, and .gif.
9. Documentation candidates including .md, .txt, .rst, .docx, and .pdf.
10. Known packaging hints from previous builds.
11. Multi-package or monorepo-like layouts where possible.
12. Service-style projects, desktop apps, CLI apps, libraries, and hybrid projects where possible.
13. Existing installer scripts or packaging manifests.
14. Data that can be transformed into a starter project JSON schema.
15. Candidate groupings by purpose, such as source, tests, assets, docs, build scripts, installers, and generated output.
16. Conflicts such as missing referenced files, duplicate icon candidates, and orphaned assets.

Project Interpretation Requirements
OtterForge must be able to build projects that do not follow a single standard layout.

Supported interpretation modes:
1. Automatic scan mode.
2. Automatic scan plus user corrections.
3. Fully schema-driven mode using a JSON project description.

The schema-driven mode must support projects with:
1. Multiple source roots.
2. Nested application folders.
3. Separate assets or resources folders.
4. Multiple executable entry points.
5. Multiple packaging outputs.
6. Platform-specific files and exclusions.
7. Custom installer resource files.

Project Organizer Requirements
OtterForge should include a project organizer that can gather files from configured source paths and arrange them into a normalized project layout according to purpose.

Organizer behavior:
1. Build an organization plan before making filesystem changes.
2. Classify each file into a target group such as source, tests, assets, splash, icons, documentation, configs, scripts, installer resources, and output.
3. Support path rules from schema section organization_rules and user overrides.
4. Preview old path to new path mappings before apply.
5. Support copy mode and move mode.
6. Write a reorganization manifest with every moved file and timestamp.
7. Support rollback by replaying the manifest in reverse.
8. Never overwrite existing files unless a force flag is provided.
9. Preserve relative references where possible by generating a post-organization report of files that may need path updates.
10. Support organizing a single project or a group of projects in one operation.

Default target folders for generated layout:
1. src/
2. tests/
3. assets/icons/
4. assets/splash/
5. docs/
6. config/
7. scripts/
8. installer/
9. dist/

Build Execution Requirements
1. Execute tools through subprocess using argument lists, not shell-joined strings.
2. Capture stdout and stderr separately.
3. Support live log streaming.
4. Persist result data after every run.
5. Support cancellation where practical.
6. Support dry-run without launching the process.
7. Support chained workflows where later stages consume prior artifacts.

Packaging and Installer Requirements
1. Packaging must be treated as a first-class workflow, not only a post-build afterthought.
2. Installer creation must consume explicit artifacts from prior steps or schema-defined inputs.
3. The system must distinguish between executable builders, package generators, and installer generators in both UI and CLI.
4. The system must record which tool produced which artifact.
5. The system must support per-platform installer strategies.
6. The system must allow raw pass-through arguments for packagers and installers the same way it does for executable builders.

Additional Feature Specifications

Pre and Post Build Hooks
OtterForge must support user-defined scripts or commands that run before and after any build, package, or installer stage. Hooks are defined per project in memory or in the project schema.

Hook types:
1. pre_build: Runs after settings are validated but before the tool command is executed.
2. post_build: Runs after the tool exits regardless of success or failure.
3. post_success: Runs only after a successful build.
4. post_failure: Runs only after a failed build.
5. pre_package, post_package, pre_installer, post_installer: Same pattern for other stages.

Hook rules:
1. A hook is a shell command string or a path to a Python script.
2. Hook failure by default halts the pipeline. A hook can be marked optional to continue on failure.
3. The hook environment receives injected variables for artifact paths, builder name, platform, and exit code from prior steps.
4. Hooks are listed and managed through otterforge hooks add, otterforge hooks list, and otterforge hooks remove.
5. The GUI must show a hook management panel where hooks can be ordered, enabled, and disabled.

Common use cases for hooks include: running tests before a build, bumping a version file, generating a changelog, notifying a webhook, uploading artifacts to a server, and cleaning temp files after a failed build.

Test Runner Gate
OtterForge must support running a test suite before any build and optionally gating the build on test results.

Behavior:
1. The user can configure a test command per project, such as pytest or python -m unittest.
2. The user can enable or disable the gate flag per project.
3. When the gate is enabled, a build will not start if the test command exits with a non-zero code.
4. The test result is stored in build history alongside the build result.
5. The CLI command is otterforge test and the gate is configured with otterforge set test.command and otterforge set test.gate true.

Code Signing
OtterForge must support signing executables and installers after they are produced.

Platform-specific signing tools:
1. Windows: signtool.exe. Requires a code signing certificate, PFX file or HSM reference, and timestamp server.
2. macOS: codesign and optionally notarytool for Apple notarization. Requires a Developer ID certificate and an Apple ID credential.
3. Linux: gpg for detached signature files of artifacts, or AppImage signing.

Signing rules:
1. Signing is a separate optional post-build stage, not part of the builder adapter.
2. Signing configuration is stored in memory but sensitive credential details such as passwords and certificate paths must never be logged or printed.
3. The user must be able to configure signing once per project and have it run automatically after every successful build.
4. The system must surface a clear warning if signing is configured but the signing tool is not available.
5. The CLI commands are otterforge sign and otterforge set signing.tool, otterforge set signing.cert, and so on.
6. Signed artifact paths replace or supplement the unsigned artifact paths in the build result.

Dependency Audit
Before building, OtterForge should optionally audit project dependencies for known vulnerabilities.

Supported audit tools:
1. pip-audit: Primary recommended tool.
2. safety: Alternative if pip-audit is not available.

Audit behavior:
1. The audit runs against the active environment or a requirements file.
2. Results are shown in the CLI and GUI with severity levels.
3. The user can configure a gate that blocks the build if any vulnerability of a given severity is found.
4. Audit results are stored in build history.
5. The CLI command is otterforge audit.

Version Management
OtterForge should be able to read, display, and optionally inject version information at build time.

Version sources in priority order:
1. Explicit value set with otterforge set version.
2. Git tag from the current commit using git describe.
3. Version field in pyproject.toml.
4. __version__ variable in the entry script or __init__.py.

Version injection:
1. The resolved version is available to the builder as the executable name suffix or as a version metadata field.
2. The version is stored in the build history record.
3. The user can optionally auto-tag the git repository after a successful build with otterforge set version.auto_tag true.

The CLI command is otterforge version show to display the resolved version and otterforge version set <value> to override it.

Icon and Asset Pipeline
OtterForge should help users prepare icon assets in the correct format for the target platform and builder without requiring manual conversion.

Behavior:
1. The user provides a source image in any common format such as PNG, SVG, or JPEG.
2. OtterForge converts it to the correct format for the selected builder and platform.
3. Windows builders require .ico format. macOS builders require .icns format. Linux builders typically accept PNG.
4. Conversion uses the Pillow library if available.
5. The converted icon is stored in a project asset cache folder and the path is written to memory.
6. The CLI command is otterforge assets prepare-icon <source_path>.
7. The GUI provides a drag-and-drop icon selector that triggers the same conversion pipeline.

Artifact Checksums and Release Manifest
After every successful build, package, or installer run, OtterForge must generate checksums and a release manifest.

Behavior:
1. Compute SHA256 checksum for every output artifact.
2. Store checksums alongside artifacts in the output directory in a file named checksums.sha256.txt.
3. Generate a release manifest file named release_manifest.json that contains artifact paths, checksums, builder name, version, platform, and timestamp.
4. The manifest can be used by CI pipelines to attach artifacts to a GitHub release or similar publishing step.
5. This behavior is on by default and can be disabled with otterforge set artifacts.checksums false.

Artifact Size Analysis
After a build completes, OtterForge should produce a size report showing what is contributing to the final executable or package size.

Behavior:
1. Walk the output directory or inspect the onefile executable structure where possible.
2. Report the top contributors by file size: Python standard library, third-party packages, and application code.
3. Where builders produce a spec or a dist folder, analyze that folder.
4. The report is printed to the console when verbose mode is active and is always written to the build history record.
5. The CLI command is otterforge artifacts size-report.
6. The GUI shows a basic size breakdown in the build results panel.

Git Integration
OtterForge should integrate with git for version reading, release tagging, and build traceability.

Behavior:
1. At build time, capture the current git commit hash, branch name, and tag if any, and store them in the build history record.
2. If version.auto_tag is enabled, create a git tag after a successful build.
3. Surface a warning if the working tree is dirty and the user is running a production build.
4. Allow the user to link a build record to a specific commit.
5. The git integration is read-only by default. Write operations such as tagging require explicit configuration.

Container and Docker Build Mode
OtterForge should support building inside a Docker container for clean, reproducible, dependency-isolated builds. This is especially useful for Linux AppImage builds and for producing builds that do not depend on the developer machine's system libraries.

Behavior:
1. The user specifies a Docker image to use for the build.
2. OtterForge mounts the project directory into the container and runs the translated build command inside it.
3. Output artifacts are written back to the host output directory.
4. The Docker image name and any extra volume mounts are stored in memory per project.
5. This mode requires Docker to be installed and available in PATH.
6. The CLI command is otterforge build --container or set with otterforge set build.container_image <image>.
7. Container mode is listed as unavailable when Docker is not detected.

Build Notifications
OtterForge should notify the user when a long-running build finishes, whether it succeeded or failed.

Behavior:
1. Use the plyer library for cross-platform desktop notifications if it is installed.
2. Fall back to a simple terminal bell or printed summary if plyer is not available.
3. Notifications include the project name, builder name, build status, and duration.
4. Notifications can be enabled or disabled with otterforge set notifications.enabled true.
5. Optional webhook notification support: the user can configure a webhook URL and OtterForge posts a JSON build result summary to it after every build.

Local Build Matrix
OtterForge should support running multiple build configurations in a single local command without requiring CI.

Behavior:
1. The user defines a matrix in the project schema or memory as a list of profiles or builder-plus-options combinations.
2. Calling otterforge matrix run executes each configuration in sequence and collects results.
3. The matrix report shows a table of which configurations passed or failed.
4. Parallel matrix execution is a future enhancement; sequential is required first.
5. Each matrix run entry produces its own build history record.

Plugin System
OtterForge must support a plugin system that allows users to add their own builder, packager, installer, or hook adapters without modifying the core codebase.

Behavior:
1. A plugin is a Python package or a folder of Python files placed in a designated plugins directory.
2. OtterForge scans the plugins directory at startup and registers any discovered adapters using the same registry mechanism as built-in adapters.
3. Plugins can implement any adapter category: builder, packager, installer, or hook.
4. The CLI command otterforge plugins list shows all installed plugins.
5. The CLI command otterforge plugins install <path_or_name> installs a plugin from a local path or from PyPI if the package follows a naming convention.
6. The CLI command otterforge plugins remove <name> removes a plugin.
7. Plugins must not be required for the core system to function. If a plugin fails to load, OtterForge logs the error and continues.
8. Document the plugin contract clearly so third parties can write compliant plugins without reading internal code.

Windows Sandbox Testing
OtterForge should support launching a Windows Sandbox session pre-loaded with a freshly built executable or installer so the user can verify it works in a clean Windows environment without affecting the host machine.

Background:
Windows Sandbox is a built-in lightweight VM available on Windows 10 and 11 Pro, Enterprise, and Education editions. It provides a disposable, isolated Windows instance that is destroyed after the session ends. It is ideal for testing installers that may write to the registry or system directories, and for confirming that executables do not have hidden dependencies on the developer machine.

Behavior:
1. After a successful build or installer creation, OtterForge can optionally open a Windows Sandbox session with the output artifact pre-copied into the sandbox.
2. OtterForge generates a Windows Sandbox configuration file (.wsb) pointing to a mapped host folder containing the artifact and an optional startup script.
3. The startup script can automatically run the installer or executable inside the sandbox so the user sees the result without manual steps.
4. The user can configure whether the sandbox session starts automatically after a build or only on explicit request via otterforge sandbox launch.
5. The sandbox configuration file is written to a temp folder and is cleaned up after the session ends.
6. OtterForge detects whether Windows Sandbox is available by checking for WindowsSandbox.exe in the system path. If it is not available, the feature is listed as unavailable with a note that it requires Windows 10 or 11 Pro, Enterprise, or Education.
7. The user can configure the startup command and any additional host folder mappings per project.
8. Read-only sandbox folder mappings are used whenever possible to prevent the sandbox from modifying host files.
9. The CLI command is otterforge sandbox launch to trigger a session with the most recent build artifact.
10. The GUI includes a launch-in-sandbox button on the build results panel that is grayed out when the feature is unavailable.

CI and Cloud Requirements
OtterForge should support GitHub-hosted or similar runners.

Required CI-related features:
1. Generate a starter GitHub Actions workflow for matrix builds.
2. Allow project profiles to declare intended target platforms.
3. Persist enough metadata so CI builds can reproduce local builds.
4. Export a profile to a portable config file.
5. Support multi-stage CI workflows that can build, package, and create installers.

Important CI note:
Do not assume that every builder is available on every runner. The system must surface missing tools clearly.

Configuration Strategy
Use explicit config files. Avoid hidden magic.

Suggested config layers:
1. Global user config.
2. Per-project config.
3. Per-profile config.
4. Per-run overrides from CLI or GUI.

Override precedence should be:
run overrides > profile config > project config > global config > adapter defaults

Acceptance Criteria For Initial Usable Version
The first usable version is complete when all items below are true:
1. A user can scan a Python project and discover likely entry points.
2. A user can select PyInstaller as the default builder without any extra configuration.
3. A user can switch to any other registered builder from the CLI or GUI without changing the underlying workflow.
4. A user can define common options and also pass arbitrary extra builder flags to any selected builder.
5. A user can save the configuration as a named profile.
6. A user can execute a build from the CLI.
7. A user can execute the same build from the PyQt UI.
8. The system stores build history and final executed command.
9. The system runs on Windows, Linux, and macOS.
10. The system correctly suppresses platform-incompatible builders based on the current host OS.
11. The system can generate a starter CI workflow for multi-platform builds.
12. A user can import or generate a JSON project description for a nonstandard project layout.
13. A user can run at least one packaging workflow through the same backend.
14. A user can run at least one installer-generation workflow through the same backend.
15. A user can run scan in files, folders, projects, and groups scope and receive classified icon, splash, and documentation candidates.
16. A user can generate and apply a project organization plan with rollback support.
17. A user can select JSON or SQL as memory backend and migrate existing data without loss.
18. An MCP client can connect and use exposed OtterForge tools with policy-controlled access.

Non-Goals For Version 1
1. Perfect support for every builder from day one.
2. Guaranteed cross-compilation to every operating system from any host.
3. Automatic installation of every external builder, compiler, or toolchain.
4. Advanced distributed build scheduling.
5. Full support for every installer ecosystem from day one.
6. Support for compiled languages beyond C, C++, Rust, and Go.
7. IDE-level debugger integration for compiled language builds.

Implementation Guidance For AI Agents
1. Reuse the existing modular command and persistent memory patterns already present elsewhere in this repository where helpful.
2. Do not hard-code a finite list of all supported tool flags.
3. Build the adapter abstraction before adding more than one builder, packager, or installer.
4. Make PyInstaller the first complete adapter. Every subsequent builder must conform to the same interface without modifications to core code.
5. Implement a registry for each category: one for builders, one for packagers, one for installers, one for language adapters. Each registry should auto-discover adapters from their respective folder.
6. Each adapter module should register itself or be discovered through the registry. Do not hard-code adapter names in the core CLI or GUI code.
7. When a builder does not support the current platform, the adapter must return False from is_available() and the registry must exclude it from listings without crashing.
8. Keep business logic out of the PyQt UI layer.
9. Prefer explicit typed models for requests, results, profiles, and stored history.
10. Keep the storage layer backend-agnostic and user-selectable between JSON and SQL.
11. Keep command execution safe by using subprocess argument lists.
12. Treat project schema ingestion as a core feature, not an optional add-on.
13. Implement the module manifest system early, before adding any optional adapters, so every subsequent adapter is governed by it from the start.
14. Every optional capability — whether a builder, installer, language adapter, service, or plugin — must register a manifest entry and must check manifest availability before running. No adapter may assume it is available without a manifest check.
15. The manifest prober must never import an optional adapter module to test availability. It must use subprocess or shutil.which to test for external tools and importlib.util.find_spec for Python packages without importing them.
16. Language adapters in otterforge/langs/ are completely isolated from the Python builder pipeline. They share only the shared services layer. Mixing Python build concepts into language adapter request models is forbidden.
17. MCP tools must be thin wrappers over existing service methods and must never duplicate orchestration logic.
18. Memory backend migrations must be implemented as idempotent, resumable operations with audit logs.
